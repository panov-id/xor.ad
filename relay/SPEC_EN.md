# relay — specification

The **relay** layer is a decentralized pool of identical nodes (the BFF / edge
tier) for the neighborhood faces. It backs the landings now (waitlist, welcome
email) and is built to host the future decentralized chat.

Companion docs: `RELEASE_{EN,RU}.md` (release & promotion regimen), `README*`
(quick start), `../docs/chat-decentralized-ideas_*` (future chat).

## 1. Topology

```
<face>.place (front on Bunny CDN)
   └─ api.<face>            entry = BFF  (the relay nodes)
         └─ [app logic] → core at xor.panov.id     (future: chat/app; N/A for the landing)
```

- One relay node serves **every brand/face** (sosed, neighbro, …). It reads the
  brand from the signup `source` (or an explicit `brand`).
- For the **landing** the node is **terminal**: `/waitlist` writes to storage +
  sends the welcome; it does not call a core. The `→ core` arrow activates with
  the app/chat.
- Public entry per face+env: `api.dev.<face>` / `api.staging.<face>` (private) and
  `api.<face>.place` (prod). These point at the pool at **cutover** (currently the
  nodes use their own `n1-<env>.relay.panov.id` hostnames behind the whitelist).

## 2. Environments & access

| Env | Where | Access | TLS | Mail | Image tag |
|-----|-------|--------|-----|------|-----------|
| **dev** | shared boxes, multi-stand | private (whitelist on 443+22) | DNS-01 | Mailpit | `dev`/branch sha |
| **staging** | same boxes, own stack | private (whitelist) | DNS-01 | Resend | release `vX.Y.Z` |
| **prod** | its own boxes | public 443 + geo-steered `api.<face>` | DNS-01 | Resend | same `vX.Y.Z` |

- **dev + staging are multi-stand** on the SAME boxes: each box runs one node
  container per env + a shared Caddy that routes by hostname
  `<box>-<env>.relay.panov.id`.
- **Private** = default-deny firewall; only 443 (from each env's `whitelist_ips`)
  and 22 (from `ssh_whitelist` + the IP the wizard connects from) are open. Port
  80 is never opened → TLS via ACME **DNS-01 (Bunny)**.
- **prod** is public on 443 (real traffic) on its own boxes; same SSH hardening.

## 3. Components (per box)

Generated by the wizard, run via `docker compose`:

- **node** (`relay-node`) — the Deno app, one container per env stack.
- **caddy** (`relay-caddy`) — shared, terminates TLS (DNS-01 via the `caddy-dns/
  bunny` plugin), routes by hostname to the right node.
- **mailpit** — dev (any env with `mail=mailpit`): catches welcome emails; UI at
  `mail-<box>.relay.panov.id`.
- **dozzle** — live container logs in the browser at `logs-<box>.relay.panov.id`.

All behind the same whitelist + DNS-01 TLS.

## 4. The node

Stateless Deno service. Identical image everywhere; only the per-env `.env` differs.

| Endpoint | Purpose |
|---|---|
| `GET /health` | liveness/readiness (reports env, storage/mail transport, brands) |
| `GET /metrics` | Prometheus counters (requests/waitlist/mail) |
| `POST /waitlist` | validate → dedup+store → per-brand welcome email |
| `POST /client-error` | fire-and-forget client error sink |
| `GET /chat` | placeholder `501` (future chat relay) |

- **Observability**: structured **JSON logs** (level/msg/fields, node, env, request
  id) and a `x-request-id` on every response; **`GET /metrics`** exposes Prometheus
  counters (`relay_requests_total`, `relay_waitlist_total`, `relay_mail_total`).
  Centralized shipping (Grafana/Loki/Prometheus) is roadmap — see `HARDENING`.

- **Storage transport** (`STORAGE_TRANSPORT`): `bunny` (Bunny Storage, one object
  `waitlist/<env>/<sha256(email)>.json`) or `fs` (a mounted dir — local stand).
  Dedup = object key by hashed email → PUT is idempotent.
- **Mail transport** (`MAIL_TRANSPORT`): `resend` (real) or `smtp` (Mailpit on
  dev/local) or `none`.
- **Welcome email**: brutalist, localized in **16 languages** (en/ru/fr/de/es/el/
  uk/be/kk/ka/hy/az/uz/ky/tg/ro; unknown → en), accent + light/dark theme.

## 5. Brands (multi-brand)

- A **brand registry** (`config.brands`): each brand =
  `{key, name, upper, domain, from, match}`.
- Default = **sosed + neighbro**. Add more (e.g. an Asia brand) via the **`BRANDS`**
  env (JSON array) — **no code change**.
- `resolveBrand(source|host)` matches against each brand's `match` list; falls back
  to the first (primary) brand. The welcome subject/body/sender are per brand;
  templates are brand-agnostic (brand token replaced at render).

## 6. DNS

- Bunny DNS zone **`relay.panov.id`** (delegated from Namecheap: `relay` NS →
  Bunny). Holds per-node records.
- Per (box, env): `<box>-<env>.relay.panov.id` → box IP. Plus `logs-<box>` and
  `mail-<box>`.
- **DNS-01 via Bunny** issues TLS with ports locked down (no public 80).
- **prod** `pool` command adds the box to the geo-steered `api.<face>` record
  (A + geolocation) — the cutover.

## 7. Images & registry

- Built in CI → **ghcr.io/panov-id/relay-node**, **relay-caddy**. Boxes
  `docker compose pull` (no on-box build → spares 1 GB free VMs + the xcaddy build).
- **Multi-arch:** `linux/amd64` + `linux/arm64` (boxes are ARM + x86).
- **Supply chain:** SBOM + SLSA provenance, **cosign** keyless signature, **Trivy**
  scan (see `HARDENING`); deps pinned via `deno.lock`.
- Tags: `:<sha>` + `:<branch>` on push, `:vX.Y.Z` on a `v*` tag. **No `:latest`**.
- Per-env **`image_tag`** in the inventory pins what each env runs.
- **Private packages:** the wizard logs the box into ghcr (`GHCR_TOKEN`,
  read:packages) before pull. If the packages are public, no token is needed
  (the images carry no secrets — config comes via env at runtime).

## 8. Provisioning & the wizard

Python wizard, runs in a Docker launchpad (`run.sh`). Inventory =
`[pool]` + `[env.*]` + `[[box]]`; secrets in `secrets.env`.

- **Box modes:** `provision` (create the VM via the provider API — Hetzner/Vultr/
  DigitalOcean) or `configure` (a box made by hand — Oracle/GCP free VMs: paste
  `ssh_host`).
- **Commands:** `status` · `provision` · `dns` · `configure` · `deploy` · `pool`
  · `up` (= provision → dns → configure).
- **configure** = harden SSH (key-only, no root, no password) + default-deny
  firewall (whitelist) + Docker + ghcr login (if private) + render & write
  `docker-compose.yml`/`Caddyfile`/per-env `.env` + `pull` + `up` + `/health`.
- **Secrets** (wizard env): `HETZNER_TOKEN`/`VULTR_API_KEY`/`DIGITALOCEAN_TOKEN`,
  `SSH_PUBLIC_KEY`, `BUNNY_API_KEY`, `BUNNY_STORAGE_ZONE/KEY`, `RESEND_API_KEY`,
  `GHCR_TOKEN?`, `WELCOME_FROM?` (global sender override — default off), `BRANDS?`.

## 9. Security

- **Firewall default-deny** per box; only whitelisted 443/22, port 80 never opened.
- **SSH hardened:** `PasswordAuthentication no`, `PermitRootLogin no`, key-only —
  use a **passphrase-protected key**. Access only from whitelisted IPs (+ the
  wizard's connecting IP, auto-allowed to avoid lockout).
- **Prod gate:** deploying a public (prod) box needs `--confirm-prod` AND the
  env's `image_tag` must be a **published GitHub Release** — the wizard verifies
  via the API. Publishing the release is the approval.
- Notes/pending: the DNS-01 `BUNNY_API_KEY` on boxes is account-wide (consider a
  scoped key); prod `/waitlist` rate-limit; geo-pool health monitor.

## 10. Release & promotion

Build-once-promote — see `RELEASE_{EN,RU}.md`. In short: dev runs the branch sha
(auto); a release is a manual `vX.Y.Z` tag built **once**; the same image goes
staging → prod; prod is gated on a **published release** + `--confirm-prod`;
rollback = redeploy the previous `vX.Y.Z`.

## 11. Testing & CI

- **Unit** (`node/`, `deno test`): email/dedup, welcome across 16 langs,
  `resolveBrand`/multi-brand.
- **Integration** (`test/integration.sh`): local stand → `/waitlist` → fs storage
  + Mailpit + dedup.
- **Smoke** (`test/smoke.sh`): post-deploy `/health` + synthetic `/waitlist`
  (run from a whitelisted host for private envs).
- **CI** (`.github/workflows/relay.yml`): tests → build+push images, on
  `relay/**` changes.

## 12. Local stand

`local/` — self-contained: node (fs storage, mail → Mailpit) + Mailpit + Dozzle.
`docker compose up`; node `:8081`, Mailpit `:8025`, Dozzle `:8090`; waitlist JSON
in `./data`.

## 13. Runbook (common tasks)

- **Add a box:** add a `[[box]]` to the inventory → `./run.sh up --node <id>`.
- **Add a brand (e.g. Asia):** set `BRANDS` in `secrets.env` (full JSON incl.
  sosed+neighbro+new) → `./run.sh deploy`. No code change.
- **Deploy dev:** push to `dev` → CI builds `:dev` → `./run.sh deploy` (or `up`).
- **Cut a release:** merge `dev`→`main`, tag `vX.Y.Z` + publish the GitHub Release
  → CI builds `:vX.Y.Z`.
- **Promote staging→prod:** bump `env.prod.image_tag` to `vX.Y.Z` →
  `./run.sh deploy --node <prod> --confirm-prod`.
- **Rollback:** set the env's `image_tag` to the previous `vX.Y.Z` → `deploy`.
- **Logs:** `logs-<box>.relay.panov.id`. **Caught mail (dev):**
  `mail-<box>.relay.panov.id`.
