# Бесплатная связка: Neon + Deno Deploy + Bunny (€0 сверх кредита)

Цель: держать бэкенд за **€0 дополнительных трат** (у тебя ~€13 кредита на
Bunny), без Supabase, и остаться портируемым (стандартный Postgres + стандартный
Deno/Web-API — съезд куда угодно без переписывания).

> Строим на текущей платформе Deno Deploy (console.deno.com) — цифры ниже под неё.

## Почему именно это под «не платить больше»

- **Always-on контейнер на Bunny жёг бы кредит 24/7** (RAM + Anycast IP ≈ $3/мес/env).
  Здесь **нет always-on своих серверов** — Neon и Deno Deploy serverless и
  засыпают → фиксированной платы нет.
- **БД не на Bunny — и это плюс:** managed Neon даёт бэкапы/ветки/scale-to-zero
  бесплатно; строка `DATABASE_URL` работает с любого хоста → портируемость
  сохранена.
- **Bunny остаётся тем, за что уже платишь** (CDN + Storage). Новых продуктов/
  подписок не заводим — тратится только существующий кредит по мере трафика
  (на альфе — центы).

## Архитектура

```
[Bunny CDN]  ── лендинги sosed/neighbro + панель  (уже так)
     │
     ▼  api.*  (proxy-зоны уже есть)
[Deno Deploy]  ── твой бэкенд (Deno/TS):
     • REST/JSON API (лендинги + панель, CORS)
     • magic-link auth (JWT через WebCrypto) + сессии
     • 3 текущие функции → роуты
     • web-push (VAPID)
     • Deno.cron: TTL-чистка эфемерки (~2ч)
     • realtime: Deno.upgradeWebSocket (лента/чат) — см. оговорку
        │
        ├─(DATABASE_URL, pooled)─► [Neon Postgres]  система записи
        ├──────────────────────►  [Bunny Storage]   стикеры/картинки
        └──────────────────────►  [Resend]          email
```

Состояние — только в Neon (и при желании Deno KV). Deno Deploy держим
**stateless** — так переживаем пересоздание инстансов.

## Что выталкивает в платно (держим в уме)

**Neon → $19:** compute >100 CU-ч/мес (БД почти не засыпает — главный тихий
триггер), данные >0.5 ГБ/проект (запись начинает падать), >10 веток, >5 ГБ egress.
**Deno Deploy → $20:** >1M запросов, **>15 CPU-часов**, >20 ГБ egress — быстрее
всего жжёт **болтливый WebSocket** (CPU-часы + egress).

Стратегия €0: редкий `Deno.cron` (раз в час, не чаще), лента — polling, чат-WS
включаем позже и следим за CPU-часами; данных мало → в 0.5 ГБ укладываемся с
запасом.

## Realtime на free — оговорка

WebSocket на Deno Deploy free **работает, но не как выделенный always-on-сервер**:
инстансы serverless, могут пересоздаваться → **клиент обязан авто-реконнектиться**,
а общее состояние держим в Neon/Deno KV, не в памяти. Для альфы/хобби-нагрузки —
ок. Если позже нужен гарантированный persistent-WS с большим fan-out — вынесем WS
отдельно. **Важно:** сейчас (вейтлист/альфа) realtime вообще не нужен — это €0 по
определению; строим WS уже под приложение.

## Портируемость (сохранена)

- **Neon** = обычный Postgres → `DATABASE_URL` работает откуда угодно; съезд =
  сменить строку.
- **Deno Deploy** = стандартный Deno + Web-API (WebCrypto, fetch, WebSocket,
  Deno.cron) → тот же код гоняется в self-hosted Deno или в контейнере.
- **Bunny** Storage за абстракцией (↔ R2/B2 меняются конфигом).
- Локально всё поднимается: `deno task dev` + локальный Postgres (или Neon-ветка).

## Плюсы сторонних (managed) сервисов

Почему Neon/Deno Deploy, а не только свой сервер:

- **Ноль ops.** Бэкапы/PITR, апгрейды, патчи безопасности, HA, мониторинг —
  на стороне провайдера. Ты не админишь Postgres и не патчишь ОС.
- **Реально €0 в простое.** Scale-to-zero: пока альфа спит — платы нет вообще.
  Свой сервер тикает 24/7 в любом случае.
- **Managed-Postgres из коробки:** point-in-time recovery, **ветки = dev/uat/prod**
  копипастом, пулинг соединений — то, что на своём боксе надо строить руками.
- **Edge/латентность.** Deno Deploy крутится ближе к пользователю; свой одиночный
  VPS — в одном регионе.
- **Меньше поверхность безопасности.** Нет своей ОС/портов/TLS-сертификатов под
  твою ответственность.
- **Быстрый старт и эластичность.** Ничего не поднимать/харденить; всплеск
  трафика провайдер держит сам, без ручного ресайза.
- **Платишь по росту.** Free-tier покрывает альфу; деньги — только когда реально
  вырастешь.

Минусы (честно): лимиты free-tier и тревога «не упереться», несколько вендоров,
меньше контроля, вендор-специфика, serverless-WS хрупче постоянного сервера,
данные лежат у провайдера.

## Альтернатива: свой дешёвый VPS (~€4/мес)

Ты прав — под такие вещи можно взять один дешёвый бокс (Hetzner CX22 ~€3.8, ARM
CAX11 ~€4, Netcup и т.п.) и держать **всё на нём Docker'ом**: свой API +
Postgres + realtime-WS + cron, все 3 окружения в одном месте.

**Плюсы VPS:**
- **Один бокс, один предсказуемый счёт** (~€4–5/мес), без лимитов free-tier и
  тревоги по CU-часам.
- **Полный контроль** — любой софт, свои версии, свои настройки.
- **Настоящий always-on Postgres и постоянный WS-сервер** (persistent-соединения,
  без serverless-реконнект-хрупкости).
- **Чистый Docker = максимум портируемости**, ноль вендор-лока.
- **Realtime без оговорок** — держит соединения сколько нужно.

**Минусы VPS:**
- **Это не €0** — ~€4–5/мес постоянно (но предсказуемо и копейки).
- **Весь ops твой:** обновления ОС, безопасность, TLS, бэкапы, мониторинг,
  админство Postgres.
- **Single point of failure** — один бокс, без HA/автоскейла.
- **Бэкапы делаешь и проверяешь сам** (`pg_dump`/снапшоты провайдера).

### Free managed vs дешёвый VPS — когда что

| Критерий | Free (Neon+Deno+Bunny) | Дешёвый VPS |
|---|---|---|
| Цена | **€0** (в лимитах) | ~€4–5/мес фикс |
| Ops | провайдер | **ты весь** |
| Бэкапы БД | managed PITR | DIY |
| Realtime WS | хрупче (serverless) | **надёжно (always-on)** |
| Лимиты | CU-часы/CPU-часы/egress | только железо бокса |
| Портируемость | высокая | **максимальная (чистый Docker)** |
| Масштаб/HA | эластично, edge | вручную, 1 регион |
| Старт | быстрее | поднять/харденить бокс |

**Как выбрать:**
- Хочешь буквально €0 и не против последить за лимитами → **free managed**.
- Готов на ~€4/мес ради полного контроля, честного realtime и «всё в одном
  Docker'е без вендоров» → **VPS**. Особенно если realtime-чат станет ядром —
  постоянный WS-сервер на боксе надёжнее serverless-free.
- **Гибрид (часто лучший):** VPS под API+realtime+cron, а **Postgres — на Neon
  free** (чтобы managed-бэкапы/ветки не строить руками). ~€4/мес и минимум ops
  на данных.

> Оба варианта используют **один и тот же портируемый образ** — начни на free
> managed, упрёшься в лимиты или захочешь надёжный WS → тем же `docker run`
> переезжаешь на VPS. Решение не запирает.

## Итог

Neon (Postgres, бэкапы, ветки, scale-to-zero) + Deno Deploy (API, auth,
web-push, cron, WS) + Bunny (CDN/Storage, уже оплачен) = **€0 сверх кредита** на
альфе, без Supabase, портируемо. Единственные «дежурства»: держать `Deno.cron`
редким, не раздувать WS и следить за CU-часами Neon — тогда в платно не
свалишься.

Если нужен полный контроль и надёжный always-on realtime — за ~€4/мес берётся
**дешёвый VPS** под тот же образ (или гибрид: VPS + Neon-free под БД). Тот же
Docker → выбор не запирает, переезжаешь `docker run`'ом в любую сторону.
