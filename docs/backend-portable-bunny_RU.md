# Портируемый бэкенд на своих контейнерах (Bunny-first, но без лока)

Направление: **свои Docker-контейнеры, которые не зависят от того, где стоят.**
Сегодня — на Bunny Magic Containers, завтра — на Hetzner/Fly/Railway/любом
`docker run`, без переписывания. Никакого Supabase-стека и никакой
Bunny-специфики (Edge Scripting, Bunny Database/libSQL) в несущем пути.

## Принцип портируемости (главное)

Всё держится на двух стандартах, которые есть везде:

1. **Стандартный Docker-образ** — твой бэкенд (API + auth + realtime + функции)
   одним образом. Запускается локально `docker compose up`, в проде — тот же
   образ.
2. **Стандартный Postgres по `DATABASE_URL`** — БД общается wire-протоколом,
   доступна с любого хоста.

Пока в критическом пути только эти два — переезд куда угодно = сменить хостинг
контейнера и (при желании) строку `DATABASE_URL`. Bunny при этом даёт CDN +
Storage + Anycast, но они **не несущие** для портируемости — их легко заменить
на R2/B2/любой CDN.

## Архитектура

```
[Bunny CDN] ── лендинги sosed/neighbro + панель (уже так)
     │
     ▼  api.*  (уже есть proxy-зоны)
[Anycast IP] ─► [Magic Containers pod: "xor-api"]  (min=max=1 на env)
                  └─ один контейнер = твой бэкенд:
                       • REST/JSON API (лендинги + панель, CORS)
                       • magic-link auth (выпуск+проверка JWT, письма через Resend)
                       • realtime WS-сервер (лента + исчезающий чат)
                       • web-push (VAPID)
                       • вызовы AI-модерации
                       • cron/интервалы: TTL-очистка эфемерки (~2ч)
                          │
                          ├─(DATABASE_URL)─► [Postgres]  (см. выбор БД ниже)
                          ├──────────────►  [Bunny Storage]  стикеры/картинки
                          └──────────────►  [Resend]  email
```

**Стек контейнера — на выбор** (любой, что любишь; все портируемы):
Deno + Hono, Node + Fastify, или Go. Deno удобен тем, что **три текущие
Edge-функции** (`invite-panel-user`, `send-waitlist-welcome`, JWT-gateway)
переезжают почти как есть — просто становятся роутами внутри контейнера.

## Выбор БД — два варианта (оба портируемы по образу, разная портируемость данных)

### Вариант 1 (рекомендую) — внешний managed Postgres, БД не привязана к месту
`DATABASE_URL` на **Neon** (free-tier, scale-to-zero, ветки = dev/UAT/prod) или
любой managed PG. Данные живут независимо от того, где крутится контейнер —
это буквально «не зависит, где стоять»: контейнер на Bunny, БД в Neon, оба
двигаются отдельно. Managed-бэкапы/PITR из коробки, не твоя забота.

- Плюс: durability, бэкапы, ветвление под 3 env, нулевая цена в простое.
- Минус: формально БД не на Bunny (но строка подключения — с любого хоста).

### Вариант 2 — Postgres тоже свой контейнер (максимально «всё на Bunny»)
`postgres:16` контейнером в **том же pod'е** (общаются по `localhost`),
persistent volume под данные. Всё на Bunny, один вендор.

- Реальность Magic Containers (проверено по докам 2026-07): том **привязан к
  одному региону/ноде, single-writer, ≤100 ГБ, 10 МБ/с, БЕЗ managed-бэкапов и
  репликации**; падение ноды может «застрять» том. → это **single point of
  failure**, бэкапы полностью на тебе.
- Митигация: `pg_dump` по cron → Bunny Storage + регулярная проверка
  восстановления; для прода перед запуском — бэкапы обязательны. Данных у нас
  мало, так что технически ок, но durability слабее Варианта 1.

> Про переносимость: образ Postgres портируем всегда; «непортируемы» именно
> **данные на томе** (привязаны к ноде/региону). Поэтому для «не зависит где
> стоять» **данные лучше в Варианте 1** (managed), а контейнер-приложение — на
> Bunny.

## Как это лежит на Magic Containers (факты, 2026-07)

- **Pod = несколько контейнеров** в общем network namespace, общаются по
  `localhost:<port>` (разные порты обязательно). Нет docker-compose внутри —
  каждый образ добавляешь контейнером в pod; порядок запуска — через
  health-checks (Startup/Readiness/Liveness), не `depends_on`.
- **Приватной сети между разными app'ами нет** — только `localhost` внутри
  одного pod'а. Поэтому если Postgres свой (Вариант 2) — он в том же pod'е.
- **WebSockets** — через Anycast IP (нужен, $2/мес/env). Лимиты
  concurrency/sticky в доках не описаны → на альфе держим **1 реплику** WS.
- **Биллинг:** CPU $0.02/CPU-час (по факту), RAM $0.005/ГБ-час блоками по 64 МБ
  (always-on пол, scale-to-zero не подтверждён), Anycast IP $2/мес, том
  $0.10/ГБ-мес. Деплой: образ в registry → dashboard/REST/**Terraform**
  (`bunnynet_compute_container_app`) — ложится в CI.

## Ориентир по цене

| | Вариант 1 (Neon) | Вариант 2 (PG в pod'е) |
|---|---|---|
| Альфа, 3 env | ~$12–18/мес (контейнер+IP ×3) + Neon $0 | ~$15–24/мес (+ тома, RAM под PG) |
| ~10k MAU | ~$30–60/мес | ~$40–90/мес |

Оба заметно дешевле ~$75 у Supabase, а главное — **без лока**.

## Что придётся написать (и что нет)

**Написать** (тонкий слой вместо «батареек» Supabase):
- magic-link: выпуск токена + письмо (Resend уже есть) + верификация → JWT.
- авторизация вместо RLS: проверки в коде роутов (или включить Postgres RLS
  самому — БД-то настоящая).
- realtime: WS-сервер (native `ws`/socket.io); эфемерка = in-memory + запись в
  Postgres, TTL-очистка cron'ом.

**Не переписывать:**
- Postgres + наши `db/migrations/*` (Вариант 1 — просто новый `DATABASE_URL`).
- 3 Deno-функции — становятся роутами контейнера.
- Bunny Storage/CDN/Resend — уже в стеке.

Оценка миграции: **~3/5** (строишь свой тонкий API/auth/realtime, но БД и логика
функций сохраняются).

## Итог

Свой один Docker-образ (API+auth+realtime+функции) на Bunny Magic Containers +
**Postgres в Neon** (данные не привязаны к месту) + Bunny Storage/CDN + Resend.
Дёшево, без Supabase, без Bunny-лока: `docker compose up` локально, тот же образ
в проде, переезд куда угодно = сменить хост и/или `DATABASE_URL`. Если нужно
буквально всё на Bunny — Postgres контейнером в том же pod'е (Вариант 2), но с
DIY-бэкапами и single-region-риском.
