# Панель — админ/модераторская панель для xor.ad

## Назначение

Внутренний интерфейс для команды: видимость и управление вейтлистом, жалобами, банами и квотами пользователей на обеих витринах (sosed.place, neighbro.place). Это операционная поверхность общего бэкенда, отдельная от пользовательских фронтендов. Панель рассчитана на рост — MVP закрывает вейтлист, авторизацию, жалобы/баны и управление пользователями панели, дальше добавляются квоты, тикеты поддержки, каталог стикеров и т.д.

## Стек

- **[Refine](https://refine.dev/)** — React-фреймворк для админ-панелей: готовый data provider и auth provider для Supabase, растущий набор плагинов — подходит под то, что панель будет разрастаться.
- **Supabase** — тот же общий бэкенд, что и у приложения (Postgres + Auth + Realtime), никакого отдельного бэкенда для панели.
- Живёт в `xor.ad/panel/` как отдельное приложение внутри репозитория гейта.

## Авторизация

Passwordless, но два разных механизма для двух разных случаев — специально не свели их в один, потому что для самостоятельного входа это была бы дыра в безопасности (см. ниже):

- **Самостоятельный вход** (уже существующий пользователь панели) — настоящий **Supabase Auth Magic Link** по email: ссылка уходит в почтовый ящик, и только у владельца ящика есть к ней доступ. Это единственное, что реально подтверждает «это ты» без пароля. Требует настоящий SMTP-провайдер — пока в `.env` плейсхолдер, письма не отправляются (см. `scripts/setup-supabase.sh`).
- **Приглашение нового пользователя** — admin вызывает Edge Function `invite-panel-user`, она создаёт ссылку через Admin API и возвращает её в ответе; admin сам копирует и отправляет её приглашённому через любой канал, которому доверяет (Telegram, личное сообщение и т.п.).

Важно: раньше здесь была версия, где ссылка для самостоятельного входа тоже просто отдавалась в ответ на HTTP-запрос по введённому email — без проверки, что запрашивающий реально владеет этим email. Это по сути отключало всю аутентификацию (любой, кто знает чей-то email, мог зайти под ним). Такой Edge Function (`request-login-link`) был удалён. Для приглашений так делать можно, потому что доверенным звеном там выступает admin, который сам выбирает канал доставки; для самостоятельного входа доверенного звена нет — только реальная доставка на владеемый email.

## Роли

- **admin** — полный доступ, включая добавление новых пользователей панели (admin или moderator).
- **moderator** — урезанные права: видит вейтлист/жалобы/баны, но не может добавлять новых пользователей панели.

Роль хранится в Postgres (таблица пользователей панели) и проверяется через RLS-политики Supabase — то есть ограничение прав на уровне базы, а не только на уровне интерфейса.

## Экраны MVP

1. **Вход** — поле email → magic link.
2. **Вейтлист** — список подписавшихся с лендингов (email, витрина-источник, дата). Обе витрины в одной таблице.
3. **Жалобы и баны** — просмотр жалоб на сообщения/пользователей, бан по UID.
4. **Пользователи панели** — список admin/moderator, форма приглашения нового пользователя по email (доступно только admin).

Квоты, тикеты поддержки и каталог стикеров — следующий этап, не в этом MVP.

## Визуальный стиль

Тот же нео-брутализм, что и на лендингах (жёсткие обводки, острые углы, тени без блюра, шрифт Unbounded для заголовков) — но акцентный цвет в самой панели нейтральный, так как она общая для обеих витрин. Красный (sosed.place) и золото/бронза (neighbro.place) используются точечно — как бейджи/полоски на строках данных, чтобы визуально различать источник записи, а не как общий акцент интерфейса.

## Открытые вопросы

- Точный SMTP-провайдер для magic link не выбран.
- Точный нейтральный акцентный цвет панели не финализирован.
- UI для управления квотами, тикетами поддержки и каталогом стикеров — не спроектирован, добавится по мере роста.
